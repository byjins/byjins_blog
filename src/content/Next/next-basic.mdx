# Next.js는 뭘까?

요즘 프론트엔드의 트렌드를 보면 `Next.js`를 상당히 많이 사용하고 있는 것을 알 수 있을겁니다. 회사가 됐던 부트캠프가 됐던 거의 빠지지 않는 필수 스택이 되었죠.

그렇다면 이 `Next.js`는 왜 많은 사람들이 사용하고 채택했을까요?

저 또한 여러 강의나 자료를 보면서 기술블로그를 만들었는데요! 아직 여러 `Next.js`의 기능을 사용하진 못했지만 과연 어떤 기능이 있고 왜 사용하는 지 알아보았습니다!

## React는 CSR?

`React`는 컴포넌트 기반 UI 라이브러리로 큰 인기를 얻었지만, 몇 가지 한계가 있었습니다.

> 이 사진은 다들 많이 보셨을 것 같습니다.

![출처: The Benefits of Server Side Rendering Over Client Side Rendering (CSR diagram)](/images/posts/Next/csr.webp)

### React의 한계점

1. SEO 문제

- React는 기본적으로 `CSR(Client-Side Rendering)`을 사용하기 때문에, 초기 로딩 시 HTML이 비어 있어서 검색 엔진이 제대로 크롤링하기 어렵습니다.

2. 초기 로딩 속도

- React 앱은 초기 로딩 시 번들 파일(js)을 모두 다운로드하고 실행해야 해서 사용자 경험(UX)이 저하될 수 있습니다.

3. 복잡한 설정

- React를 사용해 SSR(Server-Side Rendering) 또는 정적 페이지 생성을 하려면 `Express` + `ReactDOMServer` + `Hydration` 같은 추가적인 작업이 필요했습니다. 이를 위해 `Webpack`, `Babel` 설정까지 해야합니다.

## Next.js의 등장 (기술적인 배경)

`Next.js`는 2016년 Vercel(당시 ZEIT)이 개발한 `SSR(Server-Side Rendering)`과 `SSG(Static Site Generation)`를 지원하는 React 기반의 풀스택 프레임워크입니다. React 개발자들이 번거로운 설정 없이도 쉽게 서버 렌더링과 정적 사이트 생성을 할 수 있도록 만들어졌죠.

![출처: The Benefits of Server Side Rendering Over Client Side Rendering (SSR diagram)](/images/posts/Next/ssr.webp)

`Next의 기능`

1. 하이브리드 렌더링

- Next.js는 `SSR`, `SSG`, `ISR`, `CSR`을 상황에 맞게 사용할 수 있습니다.
  - 블로그 같은 정적인 페이지는 `SSG`로 빌드하여 빠른 로딩 속도를 제공.
  - 사용자마다 다른 데이터를 보여줘야 하는 대시보드는 `SSR`로 동적으로 렌더링.
  - 데이터가 자주 변경되는 페이지는 `ISR`을 이용하여 정적 페이지를 부분적으로 갱신.

3. API Routes 지원

- 서버리스 함수(Backend API)를 쉽게 만들 수 있습니다. 이를 통해 백엔드 없이도 간단한 API를 구현 가능합니다.

4. 이미지 최적화 (next/image)

- next/image 컴포넌트를 사용하면 자동으로 이미지 최적화(`Lazy` `Loading`, `WebP` 변환 등)를 수행합니다.

5. 자동 코드 분할 `(Automatic Code Splitting)`

- 필요한 페이지의 코드만 로드하여 성능을 최적화. React 앱에서는 모든 JS 파일을 한 번에 불러와야 했지만, Next.js는 페이지 단위로 분리해서 불러와 성능을 개선합니다..

6. Edge Functions 및 Middleware 지원

- Next.js 12부터 `Edge Functions`와 `Middleware`를 지원하여, 성능을 극대화하고 유저 요청을 빠르게 처리할 수 있습니다.

### 사람들이 Next.js를 사용하는 이유

위에서 본 Next.js 기능을 보면 사실 사용을 안 할 이유가 없어보입니다.

`React`의 불편함을 많이 해소해주기 때문이죠!

✅ 1) SEO 친화적

- SSR, SSG를 통해 검색 엔진이 쉽게 크롤링할 수 있는 HTML을 제공 → SEO 성능 향상.

✅ 2) 빠른 로딩 속도 (SSG & ISR)

- 정적 사이트 생성(SSG) 및 증분 정적 재생성(ISR) 덕분에 페이지 로딩 속도가 빨라짐.
- 글로벌 캐싱 및 CDN을 활용하여 성능 최적화.

✅ 3) 서버리스 API 제공

- 별도의 백엔드 없이도 API Routes를 활용하여 RESTful API 또는 GraphQL API를 만들 수 있습니다.

✅ 4) 유연한 렌더링 방식 지원

- SSR, SSG, ISR, CSR을 자유롭게 선택 가능 → 다양한 요구 사항을 충족할 수 있습니다.

✅ 5) 풀스택 개발 가능

- Next.js는 React + 백엔드(API Routes) + Edge Functions까지 지원 → 풀스택 개발이 쉽습니다.

✅ 6) TypeScript 및 CSS 지원

- 기본적으로 TypeScript, CSS-in-JS, Tailwind CSS, Sass 등을 지원하여 개발 생산성이 향상.

✅ 7) React Server Components (Next.js 13+)

- app/ 폴더에서 서버 컴포넌트와 클라이언트 컴포넌트를 분리하여 렌더링 성능 최적화.

## SSR? SSG? ISR? 그게 뭔데?

Next가 이러한 기능을 제공해준다 라는건 다들 들어보시거나 아실겁니다. 근데 거기서 `SSR`, `SSG`, `ISR`이 뭘까요?

### SSR(Server-Side Rendering)

요청(Request)이 올때마다 `서버에서 페이지를 생성`해서 HTML을 반환하는 방식.

즉, 매번 최신 데이터를 가져와서 렌더링

- 특징
  - `SEO에 좋다`: 검색 엔진이 HTML 바로 크롤링할 수 있다.
  - `최신 데이터 제공`: 매 요청마다 새로운 데이터를 받아온다.
  - `초기 로딩 속도가 느릴 수 있다` : 서버에서 HTML을 생성하는 시간이 필요하다.

> 데이터를 가져온 후, 서버에서 HTML을 생성해서 클라이언트에게 보낸다.

### SSG (Static Site Generation)

빌드 시점에 미리 HTML을 생성해서 정적인 페이지로 저장하는 방식.

- 특징
  - `속도가 빠름`: 정적인 HTML을 제공하기 때문에 로딩 속도가 매우 빠르다.
  - `SEO에 좋음`: 미리 만들어진 HTML을 제공하므로 검색 엔진이 쉽게 크롤링할 수 있다.
  - `실시간 데이터 반영 어려움`: 데이터가 바뀌면 다시 빌드를 해야 한다.

> 요청이 들어오면 이미 생성된 HTML을 바로 제공한다.

### ISR (Incremental Static Regeneration)

SSG의 단점(실시간 데이터 반영 어려움)을 보완한 방식

- 특징
  - `초기 로딩 속도가 빠름`: 정적인 HTML을 제공.
  - `최신 데이터 반영 가능`: 지정한 시간이 지나면 새로운 데이터를 가져와 갱신한다.
  - `SEO에 좋음`: 기존 SSG처럼 정적 HTML을 제공하면서 최신 데이터를 유지할 수 있다.

> 정적 페이지만 백그라운드에서 주기적으로 업데이트 가능하다.

> 상황에 비유해서 정리를 해봤습니다!
SSR (Server-Side Rendering): 매번 주문할 때마다 새롭게 음식을 준비하는 방식. 최신 정보를 반영하지만 시간이 걸린다.
SSG (Static Site Generation): 음식을 미리 준비해 두고, 고객이 오면 바로 서빙하는 방식. 빠른 속도지만 업데이트가 어렵다.
ISR (Incremental Static Regeneration): 음식을 미리 준비하면서, 정기적으로 새 음식을 준비하여 최신 상태를 유지하는 방식. 빠르고 최신 데이터를 유지.

## 마무리

`Next.js`는 단순한 프론트엔드 프레임워크가 아니라, 풀스택 개발을 지원하는 강력한 도구입니다. 많은 장점도 있고 13+에서는 App Router와 React Server Components 도입으로 성능이 더욱 향상되었죠!

`Next.js`에서 제공하는 기능을 사용한다면 기존 리엑트의 단점들을 많이 없애면서 SEO, UX 등 개발경험 및 사용자 경험을 굉장히 긍정적인 방향으로 가져갈 수 있을 것 같습니다!

> 개인적인 생각

여기저기서 다 `Next.js`를 사용하고 안하면 뒤쳐질 것 같고 그래서 모든 프로젝트에 `Next.js`를 사용해야 할까요? 저는 아니라고 생각합니다.

많은 기술을 배우고 익히되 언제나 본인 프로젝트의 규모와 필요에 따라 적합한 기술을 선택할 줄 아는 것이 더 중요하지 않을까요!?
